# SPRINT-005 Devil's Advocate Review (Blocking)

## Approval Recommendation
Reject this sprint plan as written. It looks organized, but most of the rigor is cosmetic. The document is good at specifying filenames, phases, and table columns; it is weak at proving that either command will generate trustworthy findings, bounded scope, or safe downstream work. Right now it is optimizing for process symmetry with `audit-security`, not for whether design and accessibility audits can survive contact with real repositories. Citation: [Overview](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L5), [Key Decisions](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L39), [Dependencies](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L428).

## 1) Flawed Assumptions

1. Assumes the `audit-security` workflow is "proven" enough to reuse wholesale for two fuzzier domains, when nothing in this plan demonstrates that the prior command was validated for accuracy, reviewer agreement quality, or downstream execution safety. "That sprint is complete" is being treated as "that pattern is fit for reuse." Those are not the same thing. Citation: [Overview](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L5), [Key Decisions](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L39), [Dependencies](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L431).

2. Assumes design quality is auditable through a single finding schema and a heuristic priority order, as if "design system adherence," "cross-pattern consistency," Nielsen heuristics, and platform guidance can be collapsed into one comparable signal. They cannot. Those standards frequently disagree, and the plan gives no rule for conflict resolution beyond a ranking that will hide disagreement rather than surface it. Citation: [Overview](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L10), [Finding Schemas](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L97), [P0-A Phase 2](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L157).

3. Assumes static code review plus LLM judgment is enough to produce meaningful accessibility findings for keyboard flow, focus management, screen reader support, live regions, motion handling, and cognitive load. The plan admits these areas require runtime and AT validation, then still treats the command output as an "accessibility compliance gate" and converts Level A/AA failures directly into sprint priorities. That is a contradiction. Citation: [Overview](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L15), [Inherent limitations](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L29), [Use Cases](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L54), [P0-B Phase 2](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L235).

4. Assumes a binary `Verification` field of `code` or `runtime` is enough to express confidence and required evidence for accessibility findings. It is not. A runtime issue found only by inference is not equivalent to one confirmed in-browser, and a code issue with uncertain product intent is not equivalent to a deterministic violation. This schema strips out confidence, test environment, browser/AT assumptions, and reproduction steps. Citation: [Finding Schemas](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L111), [P0-B Phase 2](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L244), [P0-B Phase 5](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L268).

5. Assumes severity can be translated cleanly into sprint priority for both domains. That is especially shaky for design, where "cost of inconsistency" is not remotely the same thing as implementation urgency, and for accessibility, where Level-based severity is only a "starting point" but is still described as a compliance gate. The plan is pretending that prioritization is mechanical when it is actually a product, legal, and engineering tradeoff. Citation: [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L86), [audit-design severity anchors](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L106), [audit-accessibility severity mapping](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L119), [Use Cases](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L54).

6. Assumes "warn and stop if no UI files found" is a sufficient guard for scope sanity. It is a weak guard. A repo can contain UI files and still be a terrible audit target because it is mostly generated code, legacy snapshots, design tokens without rendered surfaces, Storybook-only examples, or multiple apps with incompatible standards. The plan mistakes "contains UI files" for "contains an auditable product surface." Citation: [Key Decisions](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L44), [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L67), [P0-A Phase 1](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L146), [P0-B Phase 1](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L224).

## 2) Scope Risks

1. This is presented as "two new commands," but it is really a multi-surface process rollout: two command specs, two artifact directories, two finding schemas, new sprint-output conventions, cross-family consistency checks, README changes, and implicit handoff contracts with `/ledger` and `/sprint-work`. That is a lot more moving parts than the plan admits. Citation: [Key Decisions](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L39), [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L86), [P0-C](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L300), [Files Summary](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L347).

2. Scope detection is grossly underspecified for modern frontends. "HTML, CSS, JSX/TSX, Vue/Svelte/Angular components, design tokens, Storybook stories" sounds comprehensive until the repo uses MDX, server components, generated CSS, Figma token exports, native mobile views, embedded webviews, or a monorepo with multiple design systems. The plan gives no exclusion rules and no fallback when the file inventory lies about the real UI surface. Citation: [P0-A Phase 1](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L146), [P0-B Phase 1](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L224), [Definition of Done](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L384).

3. Accessibility scope is almost guaranteed to balloon because the plan includes dynamic content, live regions, focus restoration, motion preferences, and cognitive accessibility while explicitly deferring browser and AT automation. That means the command will either over-claim from static evidence or produce huge numbers of "runtime" tasks that shift the real work into the sprint. Either way, the implementation burden is being hidden. Citation: [Overview](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L29), [P0-B Phase 2](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L235), [Deferred](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L339).

4. The plan underestimates hidden dependency on design-system discoverability. `audit-design` is supposed to prioritize the project design system "if present," but there is no requirement to identify the source of truth, detect whether it is stale, or distinguish between official tokens and accidental local variations. In a repo with partial migration or no documented system, this command will devolve into pattern-guessing. Citation: [Finding Schemas](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L101), [P0-A Phase 2](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L165), [Risks & Mitigations](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L392).

5. The plan still has a sprint-number race and trivializes it as low risk. Two audit commands plus the existing family make concurrent runs more likely, not less. "Always checks current max; each audit increments" is not coordination, it is hope. Citation: [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L71), [Risks & Mitigations](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L395).

6. P1 documentation being optional is a scope risk, not a nice-to-have. If these commands ship without README and directory docs, users will invoke them without understanding static-review limits, artifact sensitivity, or the meaning of extension columns. The plan is pushing essential operator guidance out of the sprint while claiming the commands are ready to use. Citation: [Key Decisions](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L46), [P1-A](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L317), [P1-B](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L327).

## 3) Design Weaknesses

1. The architecture is too template-driven. The plan starts from "same 5-phase workflow" and forces both domains through that mold instead of asking what evidence a design audit or accessibility audit actually needs. Reuse is driving structure more than domain fit. Citation: [Key Decisions](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L39), [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L64), [P0-C](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L302).

2. Reusing the `audit-security` core schema is likely a category error. Security findings, design findings, and accessibility findings do not have the same evidence shape, remediation semantics, or confidence model. Preserving a family contract with `/sprint-work` is being prioritized over modeling the domains honestly. Citation: [Overview](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L21), [Key Decisions](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L41), [P0-C](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L311).

3. Using Codex for both the independent review and the devil's-advocate pass is still weak adversarial design. It is the same system critiquing a synthesis partly derived from its own output, now in two separate domains where subjective judgment is even harder to challenge objectively. That is not robust review separation; it is ceremony. Citation: [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L74), [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L82), [P0-A Phase 4](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L175), [P0-B Phase 4](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L260).

4. The design finding schema is too thin to capture the nuance that matters. There is no field for impacted user flow, screenshot/state context, responsive breakpoint, component variant, confidence, or whether the finding is local inconsistency versus systemic design debt. That guarantees muddy synthesis and vague sprint tasks. Citation: [Finding Schemas](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L97), [P0-A Phase 2](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L158), [P0-A Phase 5](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L181).

5. The accessibility schema is also too thin, just in a different way. It allows multiple WCAG failures to be preserved separately if remediation differs, but gives no rule for grouping failures that share one root cause or for avoiding task explosion when a single broken pattern affects dozens of components. The architecture will generate either noisy duplicate tasks or lossy rollups depending on who implements it. Citation: [P0-B Phase 3](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L252), [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L78), [Definition of Done](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L384).

6. The artifact storage design is reckless for sensitive findings. Accessibility gaps and unreleased UX critiques in versioned `docs/` are not harmless. The plan acknowledges sensitivity only as a README note, which is far too weak given that these artifacts are the primary output. If the design is wrong here, every successful run leaks review material into normal repo history. Citation: [Key Decisions](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L40), [P1-B](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L329), [Security Considerations](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L404).

## 4) Gaps in the Definition of Done

1. The DoD is dominated by structural checks: files exist, phases exist, columns exist, prompts avoid certain flags, family consistency passes. None of that proves the commands produce accurate, bounded, or useful findings. A beautifully formatted bad audit passes. Citation: [Definition of Done](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L357).

2. There is no requirement to validate design findings against an actual UI render, screenshot, or component preview. For a design audit, that is absurd. You are approving a design-review command without any acceptance criterion that checks whether the command can see the design it is judging. Citation: [Overview](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L10), [P0-A Acceptance](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L201), [Observability & Rollback](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L414).

3. There is no requirement to validate accessibility findings against real browser behavior, keyboard traversal, or screen reader output even though the plan explicitly says those limitations matter. The DoD accepts "testing guidance" in tasks instead of requiring any demonstrated accuracy for runtime-labeled findings. Citation: [Inherent limitations](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L29), [P0-B Acceptance](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L290), [Definition of Done](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L376).

4. The DoD does not require any quality threshold for synthesis. There is no acceptance criterion for deduplication correctness, false-positive handling, root-cause grouping, or whether rejected devil's-advocate challenges are persuasive instead of perfunctory. Citation: [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L78), [P0-A Phase 4](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L175), [P0-B Phase 4](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L260), [Definition of Done](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L381).

5. The DoD treats "no findings" as a valid outcome but never requires evidence that the audit had enough coverage to justify that result. A broken command that inspects the wrong files, misses rendered states, or punts everything into ambiguity can still emit a ceremonial "Verify no findings" task and pass. Citation: [P0-A Phase 5](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L185), [P0-B Phase 5](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L271), [Definition of Done](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L379).

6. There is no acceptance criterion for safe handling of sensitive audit artifacts before they are written to version control. The security section mentions sensitivity; the DoD ignores it entirely. That means a bad implementation can leak embarrassing or regulated findings and still count as complete. Citation: [Security Considerations](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L398), [Definition of Done](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L357).

7. The observability story is laughably weak. "Invoke each command on a project with known issues; confirm artifact files are produced and the sprint document is consumable" is not validation. That only proves the machinery emits documents, not that the findings are correct, complete enough, or safe to execute. Citation: [Observability & Rollback](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L414).

## 5) Most Likely Failure Mode

The most likely failure is that the commands generate polished, authoritative-looking audit sprints that teams trust more than they deserve, and those sprints convert half-validated design opinions and inferred accessibility issues into executable backlog work.

Failure chain:

1. Phase 1 scopes a repo by file type, not by real product surface, so the audit starts from an incomplete or distorted representation of the UI. Citation: [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L67), [P0-A Phase 1](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L146), [P0-B Phase 1](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L224).
2. Phase 2 produces confident findings anyway, because the schemas force crisp rows even when the evidence is partial, inferred, or missing runtime confirmation. Citation: [Finding Schemas](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L93), [P0-A Phase 2](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L157), [P0-B Phase 2](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L235).
3. Phase 3 cleans up the mess cosmetically by deduplicating and calibrating severity, which makes the output look more reliable while actually erasing uncertainty and disagreement. Citation: [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L78), [P0-A Phase 3](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L168), [P0-B Phase 3](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L252).
4. Phase 4 adds a false sense of challenge, but the adversarial pass is still another LLM review in the same process, not external validation. Citation: [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L82), [P0-A Phase 4](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L175), [P0-B Phase 4](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L260).
5. Phase 5 converts that cleaned-up uncertainty into sprint tasks and `/sprint-work` handoff, where the output gains operational authority it never earned. Citation: [Architecture](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L86), [P0-A Phase 5](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L181), [P0-B Phase 5](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L266), [Definition of Done](/Users/corey/Code/github.com/coreydaley/agent-config/docs/sprints/SPRINT-005.md#L376).

Net: the sprint succeeds administratively while failing epistemically. It will ship commands that are good at producing audit paperwork and bad at establishing when their own conclusions deserve trust.

## Blocking Verdict
Do not approve implementation until the plan proves four things it currently only asserts:

1. The commands can distinguish auditable UI scope from arbitrary frontend-shaped files.
2. The schemas preserve enough evidence, confidence, and context to support real remediation decisions.
3. The handoff to `/sprint-work` cannot launder uncertain or subjective findings into authoritative tasks.
4. The process fails loudly when runtime validation, design-system context, or artifact sensitivity make the result untrustworthy.
